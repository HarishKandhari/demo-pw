AI Voice Agent — Architecture & Design

A browser-based conversational voice agent that greets callers, answers questions from a PW Media knowledge base, and books discovery calls.

1Executive summary

The PW Media Voice Agent is a self-contained web application that behaves like a friendly front-desk receptionist named "Jordan." A visitor can speak or type; the agent understands them, answers questions about PW Media using a curated knowledge base, and collects the details needed to schedule a discovery call — then hands that lead to the team.

The system is deliberately simple and robust: it runs entirely in the browser with no application server to build, secure, or maintain. Its conversational "brain" is rule-based rather than a large language model, which makes every response instant, predictable, and strictly on-brand — it cannot go off-script or invent facts. Voice is handled by the browser's built-in speech recognition and a streaming text-to-speech service for a natural spoken reply.

At a glance

One static HTML file · No backend · Deterministic rule-based engine (no LLM) · Live speech-to-text · Low-latency streaming voice · Hosted on Netlify, deployed automatically from GitHub on every change.

2System architecture

The solution has three zones: everything that runs in the caller's browser (the whole application), the external voice service it calls, and the hosting pipeline that delivers it. There is no database or server-side code — the app is a single file served as a static asset.

2.1 Components

Speech-to-TextUses the browser's Web Speech API to transcribe the caller in real time, showing live word-by-word captions and auto-submitting when they pause. Runs hands-free: one tap starts it and it stays on for the whole call.

3Call lifecycle — one conversational turn

Every exchange follows the same predictable path. The sequence below traces a single turn from the moment the caller speaks to Jordan's spoken reply, and shows how the microphone coordinates with playback so the agent never "hears itself."

3.1 Step-by-step

  1. Capture: the caller talks; the Web Speech API streams an interim transcript to the screen and finalises it on a pause.
  2. Understand: the engine extracts any structured details (name, business, phone) and classifies the caller's intent.
  3. Decide & answer: it selects the appropriate on-brand response from the knowledge base, or advances the discovery-call intake.
  4. Speak: the reply text is sent to ElevenLabs; audio streams back and playback begins on the first chunk.
  5. Re-arm: the mic, paused during playback to avoid feedback, turns itself back on so the caller can simply keep talking.

4Conversation engine — decision logic

This is the heart of the system and the reason it feels reliable. Rather than predicting text, the engine runs each caller turn through a fixed decision tree. It always tries to answer the caller's question first, captures any useful details, and only steers toward the discovery call gently — never nagging for the phone number on every turn.

4.1 What each stage does

  • Extract — pattern rules pull structured data out of natural speech ("this is Sam from Bright Path Dental" → name = Sam, business = Bright Path Dental), with guards so ordinary questions aren't mistaken for answers.
  • Intent match — the turn is tested against ~30 question patterns: pricing, services, "are you a robot?", guarantees/ROI, clients, contracts, timelines, location, reviews, objections, and more.
  • Slot filling — the engine tracks the four discovery-call fields and knows the difference between a caller asking something and providing an answer.
  • Natural response — answers use varied wording so they don't sound copy-pasted, and the request for a callback number appears once, softly, when the caller is ready — not on every turn.

5Deployment & operations

Because the app is a single static file, delivery is trivial and effectively free. There is nothing to run — updating the agent is just editing a file and pushing it.

7Data captured & operating modes

7.1 Lead data

The agent collects four fields to qualify and schedule a discovery call:

  1. Caller's name
  2. Business / company
  3. What they want to improve
  4. Callback number

Keys for the voice service are held client-side; no caller data is stored on a server in this prototype.

7.2 Two views

A single page serves two audiences, switchable from the header:

  • Caller view — clean chat + visuals only.
  • Team view — adds a configuration panel and a per-turn QA/latency log for internal testing.

7.3 Modes

Front-desk intake is live today. Discovery Call mode — a deeper, guided call flow — is marked Coming soon in the UI.

8Design rationale

DecisionWhy
Rule-based engine (no LLM)Instant responses, zero hallucination risk, every answer pre-approved and on-brand, no per-call model cost.
No backend serverCheaper, more secure, and trivially portable — the whole product is one file that can be hosted or embedded anywhere.
Streaming voiceJordan starts speaking on the first audio chunk, making the conversation feel real-time rather than "processing…".
Config-driven knowledgeServices, answers, and intake fields are plain configuration, so new questions or a future Discovery Call mode drop straight in.

9Roadmap

  • Discovery Call mode — a longer, guided qualification flow (shown as "Coming soon").
  • Lead delivery — post captured leads to a CRM, email, or PW Media's Click2Dash platform.
  • Analytics — persist the per-turn QA log for conversation quality tracking.
  • Embed — drop the agent into pwmedia.com as a click-to-talk widget.
